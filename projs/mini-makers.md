# Mini Makers 微信小程序

如果要给我的开发之路找一个真正的起点，我会把它标在 **2025 年 2 月，大一上学期寒假**。

那时我还不是一个“会做网站的人”。我只是 Makershub 创客空间里一个想把事情做成的成员：社团需要一个更顺手的小程序入口，能承载登录注册、成员信息、活动发布、部门工作、项目和场地等日常管理场景。于是我打开微信开发者工具，从 WXML、WXSS、JavaScript 和小程序页面生命周期开始，一边查文档，一边改样式，一边在真机上调试。

这个项目不大，却很重要。它不是课堂作业，也不是只停在本地的 Demo，而是我第一次把代码对准真实组织、真实流程和真实用户。它让我第一次意识到：技术不是为了“写出来”，而是为了让一个群体的工作流变得更清楚、更轻、更可持续。

## 项目起源与故事

Makershub 是一个创客社团，日常工作天然会涉及很多分散的事项：活动宣传、任务发布、学年安排、项目跟进、物资与场地、值班签到、成员信息维护。过去这些事情更容易散落在群聊、表格和人工沟通里。

寒假里，我把它当成一次真正的产品练习：

- 先把社团成员最常用的入口放到首页，而不是一上来堆复杂后台。
- 再把“我的”和“协会工作”拆出来，让普通成员和社团管理角色有不同路径。
- 最后围绕宣传部、基管部、项目部、运维部等部门，设计可以继续扩展的二级页面。

开发过程里最有 Hacker 精神的地方，是很多问题没有标准答案：底部导航怎么适配？自定义导航栏如何处理胶囊按钮和状态栏高度？iOS 上滑动穿透怎么办？iPad 这种宽高比异常的设备怎么调？我只能不断试、不断改、不断在真机上验证。

也正是这个过程，把我从“看教程的人”推成了“能交付东西的人”。

## 核心功能

从 `app.json` 的页面配置和各页面源码来看，小程序主要包含这些模块：

### 1. 登录与注册

对应页面：`pages/login/login`、`pages/register/register`

- 邮箱登录、密码输入、记住我、忘记密码入口。
- 注册页包含邮箱、验证码、密码、确认密码等表单字段。
- 使用 `wx.request` 预留后端登录、注册、验证码接口。
- 实现密码可见性切换、输入聚焦状态、表单状态管理。

### 2. 首页功能入口

对应页面：`pages/index/index`

- 顶部轮播图与公告栏。
- 以网格按钮组织高频功能：申请借物、申请场地、项目立项、协会活动、查看项目、近期赛事。
- 底部导航连接社区、首页、我的三类主路径。
- 首页视觉使用猫猫元素、图标和青绿色品牌色，形成社团专属识别。

### 3. 我的页面与个人资料编辑

对应页面：`pages/me/me`、`pages/editPage/editPage`

- 展示姓名、联系方式、个性签名、头像、成员身份与积分。
- 支持复制联系方式 / 签名的交互入口。
- 提供我的借物、我的项目、我的场地、协会工作、荣誉墙等个人中心入口。
- 编辑页支持修改真实姓名、联系方式、个人签名、头像，并预留保存接口。

### 4. 协会工作台

对应页面：`pages/club_work/club_work`

- 常用功能：值班签到、扫除签到、远程控制、值班申请。
- 部门工作入口：宣传部、基管部、项目部、运维部。
- 每个部门使用卡片式布局，保留“进入管理”的清晰操作路径。

### 5. 宣传部工作流

对应页面：`pages/publicity_work_page/publicity_work_page`、`pages/schedule/schedule`、`pages/task/task`、`pages/activity/activity`、`pages/xiumi/xiumi`

- 学年工作安排：使用表格展示活动文案、推文、新闻稿等内容，并支持编辑状态。
- 发布任务：包含任务名称、负责人、任务内容、截止时间选择。
- 活动宣传发布：包含活动名称、海报上传、活动简介、地点、二课链接、举办时间、报名截止时间。
- 秀米链接提交：用于提交推文排版链接。
- 页面中通过 `level` 权限字段控制不同操作入口的展示。

### 6. 部门管理页面

对应页面：`pages/base_management_work_page/base_management_work_page`、`pages/project_work_page/project_work_page`、`pages/operation_maintenance_work_page/operation_maintenance_work_page`

- 为基管部、项目部、运维部保留独立管理页面。
- 通过 `wx.request` 预留用户权限 / 等级接口，为后续后端接入做准备。

## 技术栈与架构

### 微信小程序原生技术栈

项目没有依赖复杂框架，而是使用微信小程序原生能力：

- **WXML**：组织页面结构与数据绑定。
- **WXSS**：实现移动端布局、按钮、表单、卡片、底部导航和视觉风格。
- **JavaScript**：管理页面状态、表单输入、页面跳转和接口请求。
- **微信小程序 API**：使用 `wx.navigateTo`、`wx.redirectTo`、`wx.switchTab`、`wx.request`、`wx.login`、`wx.getStorageSync`、`wx.setStorageSync` 等能力。

### 页面结构

`app.json` 中注册了 14 个页面：

```text

pages/register/register
pages/index/index
pages/me/me
pages/editPage/editPage
pages/club_work/club_work
pages/publicity_work_page/publicity_work_page
pages/base_management_work_page/base_management_work_page
pages/operation_maintenance_work_page/operation_maintenance_work_page
pages/project_work_page/project_work_page
pages/task/task
pages/schedule/schedule
pages/activity/activity
pages/xiumi/xiumi
```

这些页面大致构成了“登录注册 → 首页入口 → 我的 / 协会工作 → 部门工作流”的信息架构。

### 自定义导航栏组件

项目封装了 `components/navBar`：

- 支持返回、回首页、编辑、消息等不同图标模式。
- 支持标题、背景色、文字颜色和图标主题配置。
- 通过 `navBarHeight`、`capsulePosition` 等状态适配微信小程序顶部胶囊与系统状态栏。

这是我第一次真正理解“组件化”的地方：当多个页面都需要一致的顶部导航时，应该把它抽成可配置组件，而不是每个页面复制一份。

### 移动端布局与适配

项目大量使用 `rpx`、`vw`、`vh`、Flex 布局和图片资源来适配手机端：

- 首页使用按钮网格组织功能入口。
- 我的页面使用卡片和列表组织身份、积分和个人功能。
- 宣传部页面使用纵向按钮容器承载工作流。
- 表单页使用输入框、选择器、上传区域和提交按钮组成完整操作路径。

从 Git commit 记录也能看到当时的真实调试过程：2 月 22 日针对 iOS 滑动穿透、iPad 宽高比等问题做过修复和适配。

### 后端接口预留

虽然这个仓库主要是前端，但多个页面已经预留了接口请求：

- 登录、注册、验证码接口。
- 用户信息、头像上传、个人资料保存接口。
- 活动发布、任务发布、学年安排提交接口。
- 用户权限等级接口。

这些接口地址在代码中仍是占位域名，说明这个阶段的重点是小程序前端骨架、页面路径和交互流打通，为后续后端接入留下位置。

## 我的收获与成长

### 1. 从零基础到能交付

这个项目让我第一次完整经历：看文档、拆页面、写样式、调交互、接 API、真机调试、修 Bug、合并代码。它让我明白开发不是“学完再做”，而是“边做边学”。

### 2. 第一次理解组件化

自定义 `navBar` 让我意识到，重复出现的 UI 不应该靠复制粘贴维护。组件化不仅是代码复用，更是把页面规则收敛到一个稳定接口里。

### 3. 第一次面对真实设备问题

模拟器里正常，不代表真机正常；手机正常，不代表 iPad 正常。iOS 滑动穿透、底部栏定位、宽高比适配这些问题，让我开始理解“前端工程”不只是写页面，还包括大量细节验证。

### 4. 第一次把产品思维带进代码

我开始学会问：用户打开首页第一眼应该看到什么？社团成员和管理员路径是否不同？宣传部为什么需要任务、活动、秀米链接这些独立入口？这些问题后来成为我做 Web 产品、个人网站和全栈项目时反复使用的思考方式。

### 5. 技术探索之路的起点

Mini Makers 是一个早期项目，代码里有很多稚嫩痕迹：占位接口、重复逻辑、还没完全抽象的页面状态。但它真实、热血，而且对我很关键。

它让我第一次感受到：只要愿意拆问题、查资料、动手试，就可以从一个空白模板，做出一个能被真实社团使用的产品入口。

这就是我技术探索之路的第一块砖。
